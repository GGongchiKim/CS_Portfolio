C++ 메모리·자료구조 실험 포트폴리오

C++로 직접 실험한 메모리·자료구조 학습 과정을, 대표 썸네일 3장과 전체 흐름 보기 갤러리를 한 문서에서 토글해 가며 확인할 수 있도록 정리한 포트폴리오입니다.
Intro

“머리로만 이해하지 않고, 눈으로 검증합니다.”

교과서 설명만 읽는 대신, 값 → 주소 → 구조로 사고를 확장해 가는 과정을 실제 출력과 그림으로 한 단계씩 확인했습니다. 각 실험은 먼저 대표 썸네일 3장으로 요약하고, 필요 시 전체 흐름 보기를 눌러 모든 페이지를 한 번에 확인할 수 있습니다.

포인터, 배열, 연결 리스트, 이진 트리를 직접 구현·출력해 보면서 “데이터 구조 = 값들의 모음”이 아니라 “주소와 포인터로 이어지는 메모리 위의 구조”라는 감각을 갖는 것이 목표였습니다. 이 감각은 이후 Unity/C#에서 컬렉션을 설계할 때에도 실제 메모리와 캐시, 참조 비용까지 함께 고려하는 기반이 됩니다.

실험 01

포인터: 값/주소/참조 차이의 시각적 검증

Swap 함수 3종(값·포인터·참조)으로 동일 목적(교환)을 달성하되, 데이터가 어떤 경로로 흘러가는지를 출력과 다이어그램으로 비교했습니다.

같은 “두 값을 바꾼다”는 동작이라도, 값을 복사해서 넘기는지, 주소를 넘겨 원본을 직접 바꾸는지, 참조 별칭을 사용하는지에 따라 메모리 상에서 움직이는 데이터의 위치와 비용이 달라집니다. 실험을 통해 세 방식 모두 구현해 보고, 함수 호출 전·후 메모리 상태를 출력했습니다.

  • Call by value: 함수 내부에서만 값이 바뀌고, 원본은 그대로 유지됨
  • 포인터 인자: 포인터가 가리키는 원본 메모리를 직접 수정
  • 참조 인자: 구문은 간단하지만, 결국 포인터와 동일하게 원본을 다루는 별칭

이 과정을 통해 포인터와 참조를 “문법”이 아니라 “주소를 직접 다루는 도구”로 인식하게 되었고, 이후 자료구조(리스트, 트리)를 구현할 때 주소 흐름을 더 자연스럽게 따라갈 수 있게 됐습니다.

실험 02

배열과 주소: 연속 메모리 & 주소 산술

배열 이름, 첫 원소의 주소, 포인터 변수의 관계를 출력으로 검증했습니다. (arr+i) == &arr[i]가 성립하며, 연속 메모리 특성으로 주소가 일정 간격으로 증가합니다.

실험 전에는 arr, &arr[0], int* p = arr;가 “대충 비슷한 것 같다” 정도의 이해였다면, 실제 주소를 찍어 보면서 배열 이름이 곧 첫 원소의 주소처럼 동작한다는 점과 인덱싱과 포인터 연산이 동일한 주소 산술 위에 있다는 점을 눈으로 확인했습니다.

  • 연속 메모리이기 때문에, 각 원소의 주소가 타입 크기만큼 일정하게 증가
  • arr[i]와 *(arr + i)가 완전히 같은 의미임을 출력으로 확인
  • “한 덩어리의 연속된 메모리”라는 관점에서, 배열과 포인터의 차이를 다시 정리

이 실험은 이후 캐시 친화적인 데이터 구조를 설계할 때, “연속 자료구조(배열/벡터) vs 비연속 자료구조(리스트)”를 성능 관점에서 비교하는 사고 방식의 기반이 되었습니다.

실험 03

단일 연결 리스트: 주소로 이어지는 사슬

동적할당된 노드(데이터+다음 주소)를 링크로 엮어 가변적 컬렉션을 구성했습니다. 삽입/삭제/탐색/출력의 주소 변화를 단계적으로 확인했습니다.

배열은 한 번 크기를 정하면 바꾸기 어렵다는 단점이 있습니다. 이에 비해 단일 연결 리스트는 노드를 동적으로 new로 만들고, 포인터로 다음 노드를 가리키는 사슬 구조를 사용해 필요할 때마다 앞·뒤에 노드를 붙일 수 있습니다.

  • 노드 구조: data + next 포인터
  • 삽입/삭제 시, 실제로 움직이는 것은 포인터(주소)뿐임을 그림과 출력으로 확인
  • 잘못 연결하거나 해제하지 않으면, 메모리 누수·dangling 포인터가 생길 수 있음을 체감

이 실험을 통해 “연결 리스트는 느리다/빠르다”라는 단편적인 인상이 아니라, 왜 탐색은 O(n)이고, 삽입/삭제가 특정 위치에서는 유리한지를 메모리 주소 관점에서 이해할 수 있었습니다. 또한 Unity에서 연속 자료구조(List/Array)를 우선 선택해야 하는 이유와, 정말 동적인 연결이 필요할 때만 링크 구조를 쓰는 이유를 더 설득력 있게 설명할 수 있게 됐습니다.

실험 04

이진 트리: 주소를 따라가는 분기 구조

부모가 자식을 주소 포인터로 가리키며, 순회는 주소를 재귀적으로 따라가는 과정임을 출력으로 확인했습니다.

트리는 단순한 “나무 모양 그림”이 아니라, 각 노드가 왼쪽·오른쪽 자식의 주소를 가지고 있는 포인터 구조입니다. 실험에서는 노드 구조를 정의하고, 중위/전위/후위 순회 함수를 작성해 실제 출력 순서가 어떻게 달라지는지 확인했습니다.

  • 노드 구조: data + left 포인터 + right 포인터
  • 재귀 호출을 통해, 주소를 따라 내려가며 서브트리를 방문하는 순회 방식 이해
  • 트리 구조가 검색/정렬/공간 분할(쿼드트리, BSP 등)로 확장될 수 있다는 감각 확보

이 실험을 통해 트리를 단순히 시험용 개념이 아니라, 실제 게임 개발에서 자주 등장하는 계층 구조 (예: UI 계층, 씬 그래프, 상태 트리, AI 의사결정 트리)의 기반으로 바라보게 되었습니다.
